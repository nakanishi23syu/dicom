# PACS/医用画像システム 関連用語集

本書は、医用画像管理システム（PACS）を題材にした学習用ツールを作るにあたって、
PACS・医療業界特有の用語や、クラウド型医用画像システムでよく使われる技術要素を
整理した用語集です。プログラミング初心者でも読めるよう、専門用語を噛み砕いて説明しています。

## 凡例（情報源の区別）

- **[Web]**：Web上の技術ブログ・医療系メディア・業界解説サイト等に基づく内容
- **[一般]**：PACS業界・医療業界・IT業界の一般的な知識から整理した内容
- **[推測]**：直接の裏付けはないが、技術構成・業界慣行から可能性が高いと考えられる内容

## 1. 医療・PACS業界用語

| 用語 | 一言definition | このツールでどう活きるか |
|---|---|---|
| **PACS**（パックス／Picture Archiving and Communication System） | 医療現場で撮影されたX線・CT・MRI・超音波などの医用画像をDICOM規格でデジタル化し、保存・管理・表示・共有・転送するシステム。**[一般]** | このリポジトリで作っているツール自体がPACSの簡易版にあたる。検査リスト表示・画像ビューワー・タイムラインビューなど実装している機能はすべてPACSの構成要素。**[一般]** |
| **DICOM**（ダイコム／Digital Imaging and Communication in Medicine） | 医用画像とその付随情報（患者ID・撮影条件等）を統一フォーマットで扱うための国際標準規格。画像フォーマットであると同時に通信プロトコルでもある。**[一般]** | 画像取り込み・保存・ビューワー表示のすべての土台。「非DICOM画像の取り込み」のような機能は「DICOMではないファイルをどう扱うか」という文脈で理解する必要があり、DICOMを知らないと対比が理解できない。**[一般]** |
| **モダリティ** | CT・MRI・単純X線（レントゲン）・超音波（エコー）・PETなど、画像を撮影する装置・検査種別そのものを指す医療現場用語。**[一般]** | 検査リスト・タイムラインビューは「どのモダリティで撮った検査か」を軸に情報が並ぶ。モダリティごとに画像枚数（レントゲンは数枚、CT/MRIは数百〜数千枚）が桁違いに異なる点は、ビューワーの表示性能・データ量設計に直結する。**[一般]** |
| **読影（どくえい）** | 放射線科医（読影医）がCT・MRI等の画像を見て、所見（異常の有無・部位・程度）を判断し言語化すること。撮影する技師の仕事とは別の、医師による解釈作業。**[一般]** | このツールのビューワー・タイムラインビューは読影医が使う画面を想定している。「なぜ過去検査と並べて時系列表示するのか」「なぜ画像を素早く出す必要があるのか」は読影という業務行為を理解して初めて腹落ちする。**[一般]** |
| **読影レポート** | 読影結果（所見・診断・推奨）をまとめた医療文書。主治医（依頼医）がこれを見て最終診断・治療方針を決める。**[一般]** | PACS（画像管理）は読影レポートシステムと連携する前提で作られることが多い。画像とレポートが対になって扱われる、という業務の全体像を知っておくと設計意図が理解しやすい。**[一般]** |
| **オーダリングシステム** | 医師が検査・処方・処置などを他部門（放射線科・薬剤部等）へ電子的に指示するシステム。**[Web]** | 「検査オーダー」が発行されて初めてモダリティで撮影が行われ、画像がPACSに届く。PACSより前工程にあるシステムであり、連携先の一つとして名前が出てくることが多い。**[Web]** |
| **電子カルテ（EHR/EMR）・HIS** | 診察内容・検査結果・診断・治療方針など診療情報全体を電子的に一元管理するシステム（HIS＝病院情報システムはこれらの総称）。**[Web]** | AI診断結果連携や確定レポート参照など、「他システムとの連携機能」の相手として電子カルテが出てくることが多い。PACSは画像、電子カルテは診療記録全般、という役割分担を理解しておく。**[Web]** |
| **RIS**（りす／Radiology Information System：放射線情報システム） | 放射線科内の検査予約・進捗管理・照射録管理などを担う部門システム。**[Web]** | PACS・RIS・電子カルテはよく三点セットで語られる。検査リスト表示機能はRISが持つ検査進捗情報と重なる部分があり、連携要件で名前が挙がりやすい。**[Web]** |
| **モダリティワークリスト（MWL）** | 電子カルテ・オーダリングシステム等の上位システムから、モダリティ（撮影装置）へ「次に撮るべき検査」の情報を自動配信する仕組み。**[一般]** | 電子カルテなど上位システムとの連携により、患者情報の入力自動化を実現する機能として紹介されることが多い。患者情報の手入力ミスを防ぐための重要機能。**[一般]** |
| **タイムラインビュー** | 患者ごとの検査履歴を時系列にサムネイル表示し、過去検査との比較を容易にする画面機能。**[一般]** | このツールの主要機能の一つ。読影では「前回と比べて増大/不変」という**比較読影**が重要視されるため、単なる一覧表示ではなく「経過を追う」という医療業務の必然性から生まれた機能だと理解すると設計意図が掴みやすい。**[一般]** |
| **比較読影** | 過去の検査画像と現在の画像を見比べて、病変の変化（増大・縮小・不変）を判断する読影の基本動作。**[一般]** | タイムラインビューやプリフェッチ機能（後述）が存在する理由そのもの。過去画像をいかに素早く・見やすく並べるかがPACSの価値を左右する。**[一般]** |
| **プリフェッチ**（Prefetch） | 新しい検査画像が届くのと同時に、その患者の過去検査データを自動的に施設内サーバーへ先読み・ダウンロードしておく機能。**[Web]** | クラウド上に画像を置く構成では、読影のたびに毎回クラウドから過去画像を取得すると表示が遅くなる。プリフェッチはこれを解消する仕組みで、非同期オーケストレーションと相性が良い処理の代表例。**[Web]** |
| **非DICOM画像** | docx/xlsx/pdfなどの文書ファイル、jpeg/bmpなどの一般的な画像、avi/wmv/mp4などの動画ファイルなど、DICOM規格に準拠していないファイル。**[一般]** | 「非DICOM画像の取り込み」はPACS製品でよく求められる機能の一つ。内視鏡写真や紹介状スキャン、術野動画など、DICOM画像だけでは完結しない現場のニーズに対応するためのもの。DICOMとの違いを理解していないと、なぜこの機能がわざわざ必要かが分からない。**[一般]** |
| **PDI**（Portable Data for Imaging） | IHE（医療情報連携の国際的な統合プロファイル団体）が定めた、CD-R/DVD-R/USBメモリ等の可搬型媒体でDICOM画像・関連情報をやり取りする際のフォルダ構成・データ形式の標準規格。**[Web]** | 「PDI付き紹介メディア作成」のような機能に直結する用語。転院・紹介時に画像を他院へ渡す際、施設ごとにバラバラな形式だと相手先で開けないため、PDI準拠のメディアを作ることで互換性を担保する。**[Web]** |
| **DICOMweb**（WADO-RS / QIDO-RS / STOW-RS） | 従来のDICOM通信（独自プロトコル）をHTTP/RESTベースに置き換えた次世代のDICOM通信規格群。QIDO-RS＝検索（メタデータ取得）、WADO-RS＝取得（画像本体のダウンロード）、STOW-RS＝保存（画像のアップロード）を担当。**[Web]** | クラウド型PACSでは、病院内の閉域網プロトコルではなくWebブラウザ・モバイル端末からもアクセスできる通信方式が求められるため、DICOMwebが採用される可能性が高い。GraphQL/RESTと並ぶ「画像データ専用のAPI規格」として、バックエンド実装で登場しても違和感がないように押さえておきたい。**[Web/推測]** |
| **HL7 / FHIR**（エイチエルセブン／ファイア） | HL7は医療情報システム間でメッセージをやり取りするための標準規格。FHIRはHL7の最新世代で、REST APIベースの現代的な仕様。**[一般]** | 電子カルテ・オーダリングシステムなど他システムとの連携（AI診断結果連携含む）で、患者情報や検査オーダーのやり取りに使われることが多い規格。DICOMが「画像」の標準なら、HL7/FHIRは「患者情報・診療情報」の標準、と役割分担で覚えると整理しやすい。**[一般]** |
| **3省2ガイドライン** | 厚生労働省・経済産業省・総務省が定める「医療情報を扱う情報システム・サービスの安全管理に関するガイドライン」群の通称。**[一般]** | クラウドに医用画像・患者情報を置く以上、必ず意識するセキュリティ要件。医療情報を扱うシステムでは正確・効率的な処理が必須であり、セキュリティ設計の前提知識にあたる。**[一般]** |
| **Study / Series / Instance** | DICOMデータの階層構造。Study＝1回の検査単位、Series＝撮影条件のまとまり（造影あり/なし等）、Instance＝画像1枚。**[一般]** | 検査リスト表示・画像ビューワーが扱う「検査」「画像」というデータの粒度は、この階層に対応する。DBやAPIの設計で「検査（Examination）」「画像」という概念が出てきたら、この階層のどこに当たるかを意識すると理解が早い。**[一般]** |
| **Accession Number** | 1回の検査オーダーを一意に識別する番号。RIS・PACS・レポートシステムを横断して同じ検査を紐付けるための重要なキー。**[一般]** | 複数システム連携（電子カルテ・オーダリング・PACS間）でデータを突き合わせる際の実務上のキー概念。API設計・DB設計で「検査ID」に類する項目を見たら、この概念と対応している可能性が高い。**[一般]** |
| **ゼロフットプリント（Zero footprint）ビューワー** | 専用ソフトのインストールや画像データのローカル保存をせず、Webブラウザだけで画像を閲覧できるビューワー方式。**[Web]** | 院外・移動中のモバイル環境から安全に画像参照するための方式。フロントエンドがブラウザベースである理由とも整合する、クラウドPACSらしい設計思想。**[Web]** |
| **AI解析連携（画像診断AI）** | 肺結節検出・肋骨骨折検出・所見サジェストなど、ディープラーニングを用いた画像解析結果をビューワーやレポートに反映する仕組み。**[Web]** | 画像診断AIとの連携はPACS製品でよく挙げられる機能の一つ。外部AIサービス呼び出しの非同期処理（結果待ち・リトライ）は、非同期ワークフローエンジンの典型的な適用先になりうる。**[Web/推測]** |

## 2. 技術用語（PACS開発の観点を中心に）

| 用語 | 一言definition | このツールでどう活きるか |
|---|---|---|
| **GraphQL** | クライアントが必要なデータの形を自分で指定して取得できるAPI設計方式。 | 医用画像のメタデータ（患者・検査・所見等）は項目数が非常に多く、画面ごとに必要な項目が異なる。REST APIだと画面ごとに専用エンドポイントを大量に作りがちだが、GraphQLなら1つのスキーマから必要な項目だけ取得できる。**[一般]** |
| **HotChocolate** | C#（.NET）向けのGraphQLサーバー実装ライブラリ。 | バックエンドをC#/.NETで作る場合、GraphQLサーバーの実装候補として最有力（もう一つの候補はGraphQL.NET）。医療情報という機微データを扱うため、型安全性の高いC#でスキーマを組めるメリットが活きる。**[推測]** |
| **Blazor Server と Blazor WebAssembly の違い** | Blazor Serverは画面の差分更新をサーバー側で計算しSignalR経由でブラウザに送る方式。WebAssemblyはC#コードをブラウザ内で直接実行する方式。 | 医療系・院内向けアプリではセキュリティ上の理由（機微情報をブラウザ側に持たせたくない、閉域網前提）からBlazor Serverが選ばれやすい。一方クラウド公開型で低遅延通信が難しい環境（モバイル利用）ではWebAssemblyが検討されることもある。**[一般]** |
| **JS系フレームワークとBlazorの混在構成** | 1つのシステム内でVue/React等（JavaScript系）とBlazor（C#系）の2種類のフロントエンドが併存する構成。 | 「新規機能・管理画面はモダンなJS系フレームワークで開発」「既存のC#資産・ビューワー周辺はBlazorで実装」というマイクロフロントエンド的な使い分け、または旧資産からモダン化への移行過渡期であるケースが多い。**[推測]** |
| **Temporal Workflow（Workflow / Activity / Worker）** | 長時間・多段階の処理を、サーバーが落ちても「続きから」自動再開できる形で記述・実行するワークフローオーケストレーションエンジン。 | 「画像受信→検証→保存→AI解析→ビューワー反映」のような多段パイプラインや、プリフェッチのような裏側の非同期処理は、途中失敗時に最初からやり直すと患者対応が遅れる致命的な問題になりうる。Temporalは失敗時の自動リトライ・再開を保証してくれるため、医療画像処理と相性が良いとされる。**[一般]** |
| **Saga パターン** | 複数のサービス・処理をまたぐ一連の操作が途中で失敗した場合に、それまでの処理を打ち消す（補償する）処理を組み込む設計パターン。 | AI解析サービス呼び出しなど外部連携を含む多段処理で、途中失敗時の後始末（例：仮登録したデータの削除）を体系立てて扱うための考え方。Temporal Workflowを使った実装でよく話題に上る。**[一般]** |
| **JWT認証** | ログイン時に発行される署名付きトークンをリクエストごとに送ることで、DB問い合わせなしに本人確認を行う認証方式。 | 医療情報を扱うシステムでは「誰がどの患者データにアクセスしたか」の追跡（アクセス制御・監査ログ）が求められる。JWT自体は認証の土台に過ぎず、その上に権限管理（管理者権限ユーザーのみ情報修正・削除可、等）を重ねる設計になっている点に注意。**[一般]** |
| **gRPC** | Google発の高速な内部サービス間通信プロトコル。 | Temporal WorkflowはgRPCを内部通信に使っており、.NET SDKもこれを前提とする。マイクロサービス的にAI解析サービス等と連携する場合の候補技術。**[推測]** |
| **fo-dicom** | C#（.NET）向けのDICOM処理ライブラリのデファクトスタンダード。 | バックエンドがC#である以上、DICOMファイルの解析・生成処理を自前実装せずこのライブラリに頼る可能性が高い。実際にこのリポジトリのbackendでも採用している。**[一般]** |
| **クラウド基盤とDirect Connect** | クラウド事業者のデータセンターと施設側を専用線で結ぶ、インターネットを経由しない高セキュリティな接続方式。 | 医用画像は機微な個人情報であり、3省2ガイドライン準拠のためインターネット直結ではなく専用線・VPN接続が前提となることが多い。クラウド型PACSのネットワーク構成を理解する上での基礎知識。**[推測]** |
| **DevOps / CI・CD（定期的なリリースサイクル）** | コードの変更を自動テスト・自動デプロイの仕組みに乗せ、決まった周期でリリースする開発運用の考え方。 | 医療機器プログラム（薬機法対応）に該当しうる製品は、リリースのたびに一定の検証プロセスを経る必要があり、一般的なWebサービスより慎重なリリース管理が行われる点に留意。**[一般]** |

## 3. 補足：院内設置型PACSとクラウド型PACSの違い

- **[一般]** PACS製品には大きく分けて「院内設置型」（オンプレミス、施設内にサーバーを置く）と「クラウド型」（院外のデータセンターで画像を管理し、院外からも参照可能）の2系統がある。
- **[一般]** クラウド型は中小規模の施設でも導入しやすい・保守負担が軽いというメリットがある一方、プリフェッチやゼロフットプリントビューワーなど「ネットワーク越しでも快適に使うための工夫」が製品の差別化ポイントになりやすい。
- **[一般]** 標準ビューワーにAI解析機能（読影支援）を組み込んだ製品も増えており、AI診断結果連携は現代のPACS製品によく見られる特徴の一つ。
