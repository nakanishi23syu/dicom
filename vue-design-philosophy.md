## 1. Vue が掲げる3本柱

Vue公式が「これを大事にしている」と明言している3つの目標です。

- **Approachable（親しみやすさ）** — HTML・CSS・JSの知識がある人なら、追加の勉強なしにすぐ書き始められる。Vueのテンプレート構文は、普段書くHTMLに「ちょっと機能を足しただけ」の見た目をしている。
- **Performant（性能）** — 裏側の仕組み（コンパイラ）が自動で高速化してくれるので、開発者が難しいことを意識しなくても速く動く。
- **Versatile（汎用性）** — HTMLに`<script>`タグを1個足すだけの小さな使い方から、Vite＋TypeScript＋Router＋Piniaを組み合わせたフル装備のSPAまで、同じ考え方のままスケールする。

これを「**Progressive Framework（漸進的フレームワーク）**」と呼びます。「漸進的」とは「少しずつ・段階的に」という意味で、"最初は小さく始めて、必要になったら少しずつ機能を足していける" ということです。自転車の補助輪のように、最初は最低限の機能だけで動かし、慣れてきたら本格的なツール群を追加していけるイメージです。

## 2. コア技術思想

### 2.1 テンプレート構文＋コンパイラによる最適化

Vueは基本的に`<template>`というHTMLに似た書き方を使います。これはビルド時（コンパイルの時間、つまり「仕込みの段階」）に、ブラウザが実際に実行する命令（レンダー関数）へと変換されます。

このとき、Vueのコンパイラは「ここは絶対に変化しない部分」と「ここは動く可能性がある部分」を事前に見分けます。たとえるなら、引っ越し業者が「この本棚は一切動かさない（倉庫にしまったままでいい）」「このイスだけは毎回配置を確認する必要がある」と、事前に仕分けしておくようなものです。この仕分けのおかげで、画面を更新するたびに全部の家具（DOM）を確認し直す必要がなくなり、動く可能性がある部分だけをチェックすればよくなります。これが**Virtual DOM（仮想DOM）**の差分計算を高速にする仕組みです。

一方Reactが使う**JSX**は、見た目こそHTMLに似ていますが、正体は「ただのJavaScript」です。JavaScriptは自由度が高い分、コンパイラが「ここは絶対に変わらない」と事前に見分けるのが難しくなります。そのため、Reactでは「ここは再計算しなくていいですよ」という指示を、開発者自身が`useMemo`や`useCallback`、`memo`といった道具を使って手動で伝える必要があります。

> たとえ話: Vueは「賢い秘書が予定表を見て、変更がありそうな予定だけを自動でマークしてくれる」タイプ。Reactは「本人が毎回、自分で『これは確認不要』と付箋を貼る」タイプ。

### 2.2 リアクティビティシステム（Proxyによる自動追跡）

Vue 3は**Proxy**という仕組みを使って、データの読み書きを自動的に監視します。

```js
const state = reactive({ count: 0 })
state.count++ // これだけで、このデータを使っている画面が自動で再描画される
```

Proxyは辞典で説明した「郵便物が必ず通る転送センター」のようなものです。`state.count`を読んだり書いたりするたびに、必ずこのProxyを経由するので、Vueは「誰が」「いつ」「どのデータを」使ったかを自動的に把握できます。そして、そのデータが変わった瞬間、それを使っている画面部分だけを自動で更新します。

対してReactは基本的に**イミュータブル**（辞書の「型抜きクッキー」方式）な考え方が前提です。`setState`のような専用の関数を通じて、明示的に「新しいデータに置き換えました」と宣言する必要があります。

- Vue: データに直接触って変形させる（粘土のように）→ 直感的だが、裏でProxyが常時見張っているという仕組みを理解していないと、思わぬ挙動に戸惑うことがある
- React: 新しいデータの型を毎回作り直して置き換える（クッキーの型抜きのように）→ 一手間増えるが、「いつ・どこでデータが変わったか」が追いやすい

### 2.3 SFC（Single File Component）による部品ごとのまとめ

`.vue`ファイルは、辞書で説明した「お弁当箱」の考え方です。1つの部品（コンポーネント）について、以下の3つを1つのファイルにまとめます。

- `<template>`（見た目＝HTML部分）
- `<script>`（動き＝JavaScript部分）
- `<style>`（装飾＝CSS部分）

これは「見た目・動き・装飾を、ファイルの種類ごとに分ける」のではなく「1つの機能（コンポーネント）ごとに分ける」という考え方です。さらに`<style scoped>`と書くだけで、そのCSSが「このお弁当箱の中だけで有効（スコープが効く）」になり、他の部品のデザインを誤って壊す心配がありません。

Reactにはこの「お弁当箱形式」の公式な仕組みがありません。JSXで見た目とロジックはまとめますが、CSS（装飾）は別ファイルに分けるのが主流で、その分け方も「CSS Modules」「styled-components」「Tailwind CSS」など、プロジェクトごとにバラバラです。

### 2.4 Options API と Composition API

Vueは、部品の書き方を2種類、公式にサポートしています。

| | Options API | Composition API |
| --- | --- | --- |
| 書き方 | `data`・`methods`・`computed`など、あらかじめ決まった「引き出し」に機能ごとに書き込む | `setup()`や`<script setup>`の中で、関連する処理をひとまとめに自由に書く |
| 向いている場面 | 初学者、小規模な部品 | 大規模な部品、ロジックの使い回し（Composable） |
| イメージ | 決められた項目を埋めていく「履歴書」形式 | 自由帳に自分の好きな順番でメモを書いていく形式 |

Composition APIが登場したことで、Reactの「Hooks」（`useState`や`useEffect`のような、機能ごとに再利用可能な部品）に近い書き方もできるようになりました。ただしVueは古い書き方（Options API）も切り捨てず、両方を公式にサポートし続けている点が特徴的です。

## 3. 他フレームワークとの比較

### 3.1 React との違い

| 観点 | Vue | React |
| --- | --- | --- |
| 見た目の書き方 | HTMLに似たテンプレート | JSX（HTMLの皮をかぶったJavaScript） |
| データの変え方 | Proxyが自動で見張る（ミュータブルでOK） | 明示的に新しいデータへ置き換える（イミュータブル前提） |
| 速度の最適化 | コンパイラが自動でやってくれる | 開発者が`memo`などを使って手動でやる |
| 装飾（CSS）の扱い | お弁当箱（SFC）に公式に同梱、`scoped`で範囲を絞れる | プロジェクトごとに方式が違う（統一の公式仕様なし） |
| 根底の考え方 | 「専用の仕組み（テンプレートやProxy）を用意して、開発者の負担を減らす」 | 「特別な仕組みはなるべく使わず、全部ただのJavaScript関数として透明にする」 |

Reactは"UIはただの関数"というシンプルさ・透明性を大事にする考え方です。Vueは逆に、専用の道具立て（テンプレート・自動リアクティビティ）を用意して、開発者が書く量そのものを減らすことを優先しています。

### 3.2 Angular との違い

| 観点 | Vue | Angular |
| --- | --- | --- |
| 導入の仕方 | 小さく始めて、必要な分だけ機能を足せる | 最初からほぼ全部入りの「大型フレームワーク」を一括導入する |
| 使う言語 | JavaScript・TypeScriptどちらでもOK | TypeScriptがほぼ必須 |
| DI（依存性注入） | 基本的になし（Composableという小さな関数で代用） | 標準機能として組み込まれている |
| 非同期の扱い | 普通のPromise（`async/await`）が基本 | RxJS（Observable、辞書の「川の流れ」の仕組み）が標準 |

Angularは「大人数のチームが同じルールで開発できるようにする」ことを重視した、規約がっちりめのフレームワークです。Vueは真逆で、「最初は身軽に、あとから必要な分だけ足す」ことを大事にしています。

### 3.3 Svelte との違い

| 観点 | Vue | Svelte |
| --- | --- | --- |
| Virtual DOM（下書き設計図） | 使う（下書きを作ってから本番DOMに反映） | **使わない**（コンパイルの段階で、直接本番のDOM操作コードを作ってしまう） |
| 実行中に必要な仕組み（ランタイム） | コンパイラ＋軽量な仕組みの両方が必要 | コンパイラでほぼ全部済ませ、実行中の仕組みはほぼ不要 |
| 共通点 | どちらも「コンパイラが賢く最適化してくれる」という考え方 | 同じ |

Svelteは、Vueの「コンパイラが最適化してくれる」という考え方をさらに突き詰めたフレームワークです。Vueが「下書き（Virtual DOM）を作ってから本番に反映する」のに対し、Svelteは「下書きすら作らず、最初から本番用の操作コードを直接作ってしまう」というアプローチを取ります。

## 4. まとめ

Vueの設計思想を一言でまとめると、

> **「普段のHTML・JS・CSSに近い書き味を保ちながら、裏側（コンパイラとリアクティビティ）を賢く最適化することで、小さな導入から大規模開発まで、同じ考え方のままスケールできるようにする」**

というフレームワークです。

- **Reactに対して**: 「全部素のJavaScriptで透明に」ではなく、「専用の道具立て（テンプレート・自動リアクティビティ）で書く量を減らす」ことを選んでいる
- **Angularに対して**: 「最初から全部入り」ではなく、「小さく始めて必要な分だけ足す」ことを選んでいる
- **Svelteに対して**: 「コンパイラが最適化する」という考え方は共有しつつ、Virtual DOM（下書き設計図）を残すことで、実行中の柔軟さを確保している
